## 構造的な問題（サンプル外で発見・重要）

### フィールドずれによるデータ破損が約30件以上存在
`lastChecked` フィールドの値を全件集計したところ、日付以外の値を持つレコードが多数見つかった:

- `lastChecked:'TRUE'` … 14件 / `'FALSE'` … 3件
- `lastChecked` にURLが入っているもの … 9件以上
- `lastChecked` に担当課名・電話番号が入っているもの … 6件以上

**原因**: 金額にカンマを含む値（例「13,000円」「379,000円」）がCSV→JS変換時に列ずれを起こしたものと推定。amount/condition/desc/period/contact/applicationUrl が1〜2列ずつずれている。

**影響範囲**: 主に id:961〜1016（東京都多摩・23区東部）、id:1227〜1246（神奈川県 小田原・茅ヶ崎・厚木・大和・海老名）、id:1396（福生市）など。これらは applicationUrl フィールドに正しいURLが入っておらず、**アプリ上で申請リンクが機能していない**。

- **URL生存率**: 24件中21件（88%）。404の3件もうち2件は市サイトのURL再編が原因で、制度自体は存続。

- **重大エラー**: 鹿児島市の「上限50万円→実際10万円」のような金額過大表示は、ユーザーの信頼を直接損なうため最優先で修正すべき。

2. **今回判明した6件の個別修正**（id:101, 226, 961, 2042, 2294, 2361, 2489, 2504 / 697は削除）。
3. **鹿児島県・奈良県など2026-05-11付け一括収集分の金額再確認** — 公式ページの「事業所向け」と「個人向け」の数字を取り違えるパターンが2件確認されたため、同バッチ（lastChecked:'2026-05-11' 974件）のうち「上限○○万円」系は抜き取り再検証を推奨。
4. **臨時給付金（物価高対応系）の期限監視** — id:697型の年度限り給付は期限到来後の自動非表示ロジック（periodの日付パース）をアプリ側に実装推奨。
