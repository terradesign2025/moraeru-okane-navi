## 1. 会議サマリー（結論3行）

1. **プロダクトの骨格（3ステップ診断×移住特集×1市区1社スポンサー）は強い**が、「278市区/1,308件」「279市区/1,428件」「3,000以上」と数字が4種類混在しており、信頼性とAIO評価を自ら毀損している状態と考えます。
2. **収益導線が未開通**：sponsor_lpのStripeリンクが全てテストモード、Web3Formsもテスト用キーのまま。スポンサー申込が来ても決済できないため、営業開始前に最優先で差し替えが必要です。
3. データは「3重管理のALL_CITIES」「id重複の暗黙排除」、実装は「レンダー内コンポーネント定義」「1,720行単一ファイル」が技術的負債の中核。**今週はS項目5件（数字統一・決済本番化・コンポーネント外出し・データ一元化・高齢者向け文字サイズ）に集中**することを推奨します。

## 2. 各メンバーの指摘

### 2-1. RADAR（戦略・市場分析・収益化）

**良い点**

1. **広告商品の設計が秀逸**：1市区1社独占×年額固定（¥120,000/¥216,000/¥306,000）は説明が一言で済み、sponsor_lpのクリック課金比較（¥300〜800/クリック→年36〜96万円）による価格アンカリングも効果的。スポンサー不在期間も「お金のプロ」3社PR枠で収益面の空白を作らない設計は他アプリ（休日ドクターナビ等）へ横展開できる型です。
2. **「住んでいない場所でももらえる」移住支援金特集は競合にない独自フック**。自治体公式サイトもYahoo!くらしも「居住地ベース」でしか見せないため、明確な差別化軸。ブログ記事→LP→アプリの導線も既に敷かれています。

**改善点**

| 優先度 | 指摘 |
|---|---|
| **S** | **Stripe決済リンクが全てテストモード**（sponsor_lp.html L1161-1163: `buy.stripe.com/test_...`、L491/526/561の`checkout.html?plan=`も同様）。申込が来ても入金できない。Web3Formsキー（L1035: `8b664d19-...`）もテスト用。本番キー発行ルール（クライアント別キー発行）に従い差し替えが必要。 |
| **S** | **掲載数字が媒体ごとにバラバラ**：lp.html統計バー「278市区/1,308件」（L158/162）、ヒーローバッジ「279市区/1,428件」（L143）、アプリ内ヒーロー「全国3,000以上の補助金」（App.jsx L438）、ヘッダー「279市区・1428件」（L391）。広告主が検証した瞬間に信頼を失うリスク。 |
| **A** | **SEO/AIO流入の受け皿が無い**：lp.htmlにJSON-LD（FAQPage/WebApplication/Organization）・canonical・og:image・FAQセクションが一切無し。GA4もコメントアウトのままで効果測定不能（→スポンサーへの月次レポート提供もできない）。 |

| **B** | 市区別ランディングページ（例: `/city/hachioji.html`）が無く「八王子市 補助金」等のロングテール検索が取れない。all_subsidies.jsから静的生成するスクリプトを次フェーズで検討する価値あり（粗大ごみアプリ全国展開と同じ戦略）。 |

### 2-2. CORE（データ・バックエンド・更新運用）

**良い点**

1. **CSV→生成スクリプトの半自動パイプラインが確立済み**：`data/`配下のセッション別CSV（32ファイル）→`scripts/csv_to_all_js.mjs`→`all_subsidies.js`の流れと、`verification/`フォルダ（検証CSV/HTML・未収録市区リスト）による検証文化は全国展開の基盤として評価できます。
2. **データ鮮度のスキーマ設計**：`verified`/`lastChecked`/`applicationUrl`フィールド、`[終了]`/`[期限確認中]`プレフィックス規約により、UI側で「公式確認済み」「受付終了」を出し分けられている点は競合ポータルより誠実。

**改善点**

| 優先度 | 指摘 |
|---|---|
| **S** | **ALL_CITIESが3重管理**：①master.js手動配列（L51-238）②`KANTO_CITIES`スプレッド③all_subsidies.jsの`ALL_SUBSIDY_CITIES`。しかもmaster.js内だけでも**名古屋市・豊田市・岡崎市がL64-66とL110-112で二重定義**。検索オートコンプリートに同じ市が2回表示される実害が出ます（App.jsx L222はdedupeしていない）。 |
| **S** | **id重複の暗黙排除が事故の温床**：SUBSIDIES末尾の`.filter(seen)`（master.js L561）は重複idを黙って捨てる。master.js内Phase2配列とall_subsidies.jsが同じid帯（201〜）を使っており、「どちらの情報が表示されるか」が配列の順番依存。データ更新時に古い情報が勝つ事故が起こり得ます。ビルド時のid重複検知が必要。 |
| **A** | **deadline欠落で締切アラートが機能しない**：master.js手書き分には`deadline`（ISO形式）があるが、all_subsidies.js由来の大半は`period`（和暦文字列）のみ。`isDeadlineSoon()`はdeadline頼みのため、1,428件の大部分で「まもなく締切」表示が動作しません。CSVに`deadline`列を追加し生成スクリプトで正規化を。 |
| **A** | **自動生成のeligible/howが全件同一の汎用文**（master.js L556-557「住民票があること」等）。制度ごとに所得制限・年齢制限が違うため誤案内リスク。「※一般的な流れです。正確な要件は公式サイトで」の注記をUI側に明示するか、未取得の場合は項目自体を非表示に。 |
| **B** | `CITY_RECYCLERS`という変数名が粗大ごみアプリ時代の遺産（コメントにも2026-05-15移行と記載）。`CITY_SPONSORS`等へ改名し認知負荷を下げる。META.lastUpdated（5-14）とall_subsidies生成日（6-7）のズレも更新運用ドキュメント（毎月27日更新ルール含む）に一本化を。 |

### 2-3. PIXEL（UI/UX・アクセシビリティ）

**良い点**

1. **エリア→条件→結果の3ステップ＋ステップインジケータ＋GPS自動特定**は迷いのない王道フロー。地方→都道府県→市区の段階選択、「データあり/準備中」バッジによる期待値管理、準備中市区を選んだ際の丁寧なメッセージ（L290）も好印象です。
2. **情報の鮮度・状態の可視化が丁寧**：「受付終了」「期限確認中」「公式サイトで確認済み」「参考情報」バッジ、締切1ヶ月以内の赤色警告（`isDeadlineSoon`）は、補助金という鮮度命の情報で大きな信頼要素。

**改善点**

| 優先度 | 指摘 |
|---|---|
| **S** | **文字サイズが高齢者ターゲットに対し致命的に小さい**：`text-[9px]`〜`text-[11px]`がアプリ全体で多用（トピックカード市名9px・スポンサー枠PR表記9-10px・通知メタ10px等）。さらに本文の`text-gray-400`は白背景でWCAG AAコントラスト比4.5:1未達。最低ラインとして本文12px・重要情報（金額/締切）14px以上、補足テキストはgray-500以上へ。 |
| **A** | **モック機能が「本物のふり」をしている**：田中美咲のアカウント、通知5件、登録中アラート3件、パスワード変更欄、アラート登録フォーム（送信先が無く`setView('success')`のみ）。高齢者ほど「登録したのに通知が来ない」と混乱します。バックエンド接続までは登録系を「準備中」表記にするか、GAS+スプレッドシート（KOMEKO NOTE方式）で実体化を。 |
| **A** | **利用規約・プライバシーポリシーが登録画面でリンクになっていない**（App.jsx L1458はただのspan）。privacyビューは存在するので`onClick`で繋ぐだけ。利用規約ビュー自体は未実装のため新設が必要。 |
| **A** | **アクセシビリティ基礎の欠落**：通知ベル・アバターボタンに`aria-label`無し、条件チェックボックスは`sr-only`だがフォーカスリング無し、診断結果0件時の空状態UI無し（0件だとスポンサー枠だけが並び「広告アプリ」に見える）。0件時は「条件を緩めて再検索」「全件見る」の救済導線を。 |
| **B** | `max-w-sm`中央寄せのためPCでは左右が真っ白。LP（緑系）とアプリ（青系）のブランドカラー不一致も、LP→アプリ遷移時に別サービス感を生むため、いずれか（緑推奨。「もらえる」=お金=緑）に統一を検討。 |

### S-2. 決済・フォームの本番化【収益直結】

**対象**: sponsor_lp.html L1035（Web3Formsキー）、L1161-1163＋L1207（Stripeテストリンク4箇所）、checkout.html

**実装方法**:
1. Stripeダッシュボードで本番Payment Link 3本（¥120,000/¥216,000/¥306,000）を発行し、`STRIPE_LINKS`オブジェクトとフォールバックURL（L1207）を差し替え。

### S-3. レンダー内コンポーネント定義の外出し【パフォーマンス・バグ予防】

**対象**: App.jsx `GlobalSponsorBlock`/`Back`/`Steps`/`TabBar`/`TopicCard`

**実装方法**: App関数の外（またはコンポーネントファイル）へ移動し、クロージャで掴んでいた値をpropsに変換:
```jsx
// Before（App内・毎レンダー再生成）
const GlobalSponsorBlock = ({ compact }) => { ...currentSponsor, sponsorCity... };
// After（App外）
function GlobalSponsorBlock({ compact, sponsor, sponsorCity }) { ... }
// 呼び出し側: <GlobalSponsorBlock sponsor={currentSponsor} sponsorCity={sponsorCity} />
```
`Back`は`onBack={()=>setView(to)}`/`onHome`をprops化。あわせて`filteredSubsidies`を`useMemo`でラップ。

### S-4. 都市マスター一元化＋id重複検知【データ品質】

**対象**: master.js L51-238（手動ALL_CITIES）、L561（暗黙dedupe）

**実装方法**:
1. `ALL_CITIES`を`ALL_SUBSIDY_CITIES`（all_subsidies.js）からの自動生成に一本化。「準備中（hasData:false）」表示用の未収録市区は別配列`PLANNED_CITIES`としてCSV管理し、`[...ALL_SUBSIDY_CITIES, ...PLANNED_CITIES]`をキー`pref:city`でdedupeしてexport。
2. `csv_to_all_js.mjs`に重複id検知を追加:
   ```js
   const dup = ids.filter((id, i) => ids.indexOf(id) !== i);
   if (dup.length) { console.error('重複ID:', dup); process.exit(1); }
   ```
3. App.jsxのサジェスト（L222）にも`new Map`方式のdedupeを適用（恒久対策が入るまでの保険）。

### S-5. 高齢者向け文字サイズ・コントラスト改善【ターゲット適合】

**対象**: App.jsx全体の`text-[9px]`/`text-[10px]`/`text-gray-400`本文

**実装方法**: 置換方針を決めて一括修正——`text-[9px]`→`text-[11px]`、`text-[10px]`→`text-xs`(12px)、本文系`text-gray-400`→`text-gray-500`以上。金額・締切・CTAは14px以上を確保。修正後にChromeのLighthouse（Accessibility）でコントラスト検査。装飾的なPRバッジ等は現状維持可だが、制度名・金額・期限・問い合わせ先の4情報は必ず基準クリアを。
